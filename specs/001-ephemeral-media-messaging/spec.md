# Feature: Ephemeral Media Messaging (Backend)

## Summary
Build a privacy-first, media-centric messaging backend that acts as a "Blind Broker" for 1-on-1 end-to-end encrypted (E2EE) photo sharing. The server functions as a secure, ephemeral pipe: it routes encrypted packets between users but retains no knowledge of message content or "read" status. Message artifacts are deleted from the server immediately upon successful delivery (ACK) to the recipient.

## Background and Goals
Users want the spontaneity of ephemeral messaging with absolute privacy. This backend MVP focuses on secure 1-on-1 photo messages. Unlike traditional chat servers, this system refuses to track "read" status or keep history. It is a "Zero-Knowledge" system where the server handles encrypted blobs and deletes them as soon as they are handed off.

## Clarifications

### Session 2025-12-17 (Refined 2025-12-26)
- Q: How does a user authenticate? → A: Username/password (Signal-style) for session tokens. Device requests are signed with a private key.
- Q: Receipt tracking? → A: **None.** The server does not track "Delivered" or "Read" states. Receipts are generated by the client and sent as standard encrypted messages back to the sender. The server cannot distinguish a "Read Receipt" from a "Photo Message".
- Q: When is media deleted? → A: **Burn-on-Delivery.** The moment the recipient acknowledges receipt (ACKs) of the message, the server deletes the encrypted blob and the message record. The *Client App* is responsible for enforcing "Burn-on-View" (deleting the local file after the user sees it).
- Q: Data retention? → A: Max 30 days. If a message is not ACKed within 30 days, it is hard-deleted. No intermediate retention.

## Actors
- **User**: Human with a unique username.
- **Device**: Client instance holding keys. Enforces "Burn-on-View" policy locally.
- **Server**: Blind Broker. Stores encrypted blobs only until delivery.
- **Admin**: Infrastructure operator. No access to content or interaction history.

## User Scenarios & Testing
- **S1 — Blind Routing**: Alice sends a message to Bob. Server stores blob and metadata. Server pushes notification to Bob. Bob connects, downloads, and ACKs. Server immediately deletes all records.
- **S2 — Client-Side Receipts**: Bob opens the message. Bob's client sends an encrypted "Read Receipt" message to Alice. Server routes this blind packet to Alice. Alice's client decrypts it and updates UI.
- **S3 — Offline Expiry**: Alice sends to offline Bob. Server holds blob. Bob never connects. After 30 days, server deletes blob and metadata.

## Functional Requirements (testable)
1. **Unique Usernames**: Users register a unique `username` (3–30 characters).
2. **Account & Device Keys**: Server stores only public identity keys.
3. **End-to-end Encryption (E2EE)**: Server stores only opaque ciphertext.
4. **Burn-on-Delivery**: Upon receiving a deletion request (ACK) from the recipient, the server must immediately and irreversibly delete the message record and the associated ciphertext blob. Tests: Fetching message/blob after ACK returns 404.
5. **Real-time Notification**: Server pushes `new_message` events via WebSocket.
6. **Blind Receipt Routing**: The server treats receipts (delivered/read) as standard encrypted messages. It does not validate or track receipt status.
7. **Message Expiry**: Un-ACKed messages are hard-deleted after 30 days.
8. **Zero-Knowledge Metadata**: Server stores minimal routing data (Sender, Recipient, Timestamp). No logs of "who read what".
9. **Abuse Mitigation**: Rate limits on uploads.
10. **Audit-Free**: No history retained post-delivery.

## Success Criteria
- **Delivery latency**: <2s for online users.
- **Cleanup reliability**: 100% of ACKed messages are removed from storage within 1 second.
- **Zero-knowledge**: Server has no code paths to track "Read" status.

## Key Entities
- `User` (id, username, public_keys)
- `Message` (id, sender, recipient, blob_pointer, created_at)
- `CiphertextBlob` (id, s3_key, size, checksum)

## Assumptions
- **Client Trust**: We rely on the Client App to honestly execute "Burn-on-View" (delete local file after display). The Server cannot enforce this on the device, only on its own storage.
- **Blind Pipe**: The server does not know if a message is a photo, text, or a receipt.

## Non-Goals
- Server-side read receipts (impossible by design).
- Message history / cloud backup.
- Multi-device synchronization (MVP is single-device).

## Acceptance Criteria
- AC1: Alice sends to Bob. Bob ACKs. Server DB is empty. S3 bucket is empty (minus orphans).
- AC2: Alice sends to Bob. Bob reads it. Bob's client sends receipt to Alice. Alice receives receipt. Server logs show 2 opaque messages delivered.
- AC3: Message sits for 31 days. Sweeper deletes it.
