# Data Model: Ephemeral Media Messaging (Revised)

## Core Philosophy
To maintain a "Blind Broker" architecture compliant with Signal protocol best practices:
1.  **No Refernecial Integrity between Messages and Media:** The database does not know which media file belongs to which message. The link is encrypted inside the message payload.
2.  **Opaque Payloads:** Message bodies are stored as raw binary blobs (`BYTEA`).
3.  **Queue-Based Messaging:** The `Messages` table is a queue. There is no `read` or `delivered` status column. Rows are deleted upon client acknowledgement.

## Tables

### 1. Users
Represents the account identity. Authentication is separate from encryption keys.

| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | UUID | PK | Unique User ID. |
| `username` | VARCHAR(50) | UNIQUE, NOT NULL | Public handle. |
| `password_hash` | VARCHAR(255) | NOT NULL | Argon2id hash for authentication. |
| `identity_key` | BYTEA | NOT NULL | Long-term public Identity Key (IK) for X3DH. |
| `created_at` | TIMESTAMP | DEFAULT NOW() | Account creation time. |

### 2. Devices
Supports multi-device capability (e.g., Mobile + Desktop). Messages are targeted at specific devices, not just users.

| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | UUID | PK | Unique Device ID. |
| `user_id` | UUID | FK -> Users.id | Owner of the device. |
| `registration_id` | INTEGER | NOT NULL | ID generated by the client during install. |
| `last_seen` | TIMESTAMP | | Used to prune abandoned devices. |
| `enabled` | BOOLEAN | DEFAULT TRUE | |

### 3. SignedPreKeys
Medium-term keys rotated periodically by the client (e.g., weekly).

| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | UUID | PK | |
| `device_id` | UUID | FK -> Devices.id | |
| `key_id` | INTEGER | NOT NULL | Client-assigned ID. |
| `public_key` | BYTEA | NOT NULL | The public key data. |
| `signature` | BYTEA | NOT NULL | Signature of this key by the Identity Key. |
| `created_at` | TIMESTAMP | DEFAULT NOW() | |

### 4. OneTimePreKeys
A pool of keys refilled by the client. These are deleted immediately after being fetched by a sender.

| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | UUID | PK | |
| `device_id` | UUID | FK -> Devices.id | |
| `key_id` | INTEGER | NOT NULL | Client-assigned ID. |
| `public_key` | BYTEA | NOT NULL | The public key data. |

### 5. Messages
A temporary queue for undelivered messages.
**Critical:** There is no `media_id` here. The payload is opaque.

| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | UUID | PK | |
| `destination_device_id` | UUID | FK -> Devices.id | The specific device queue this message is for. |
| `source_user_id` | UUID | FK -> Users.id | Who sent it (for rate limiting/blocking). |
| `type` | INTEGER | NOT NULL | Protocol Buffer type (e.g., 1=Ciphertext, 2=PreKeyBundle). |
| `body` | BYTEA | NOT NULL | **Encrypted** Protocol Buffer envelope. |
| `timestamp` | BIGINT | NOT NULL | Client-provided timestamp (msg generation). |
| `server_timestamp` | BIGINT | DEFAULT NOW() | When it hit the server (for TTL vacuuming). |

### 6. Media
A dumb storage bucket for encrypted blobs.
**Critical:** No `uploader_user_id` is recorded to prevent traffic analysis linking users to files.

| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | UUID | PK | Random ID generated by server on upload. |
| `file_path` | VARCHAR(255) | NOT NULL | Path to object storage (S3/Local). |
| `content_type` | VARCHAR(50) | NOT NULL | MIME type (e.g., application/octet-stream). |
| `size_bytes` | BIGINT | NOT NULL | For quota enforcement. |
| `created_at` | TIMESTAMP | DEFAULT NOW() | |
| `expires_at` | TIMESTAMP | NOT NULL | Hard deletion time (TTL). |

## Entity Relationships
1.  **User** 1 -- * **Device**
2.  **Device** 1 -- * **SignedPreKey**
3.  **Device** 1 -- * **OneTimePreKey**
4.  **Device** 1 -- * **Message** (Queue)
5.  **Media** is isolated. It is only referenced inside the encrypted `body` of a `Message`.
